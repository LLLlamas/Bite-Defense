# Bite Defense вҖ” Native iOS Port Plan (Swift + SpriteKit)

**Goal:** ship Bite Defense as a fully native iOS app built in Swift with SpriteKit for rendering and game logic. Target TestFlight first, then public App Store release. The existing JS/Canvas version becomes a reference implementation вҖ” design, balance, and feel stay, but all code is rewritten natively.

**Why native over Capacitor:** tighter 60/120fps rendering, real Metal acceleration, native haptics/animations with zero bridge latency, smaller binary, better battery life, easier App Store approval ("real" apps aren't scrutinized for minimal functionality), and a foundation for future features like Game Center, iCloud sync, and iPad-optimized layouts.

**Apple Developer Program:** вң… enrolled.

**Reference implementation:** the current JS codebase in `src/` stays in-repo as the source of truth for game design, balance numbers, and UX decisions. No functional change to it during the port.

## 1. Technology Stack

| Layer | Choice | Why |
|---|---|---|
| Language | **Swift 5.9+** | Apple's first-class language; great tooling |
| Game engine | **SpriteKit** | Apple-native 2D engine, perfect for tile-based isometric games, built-in physics and actions, zero dependencies |
| UI chrome (HUD, panels, modals) | **SwiftUI** | Modern declarative UI, composes cleanly with SpriteKit via `SpriteView` |
| Persistence (local) | **Core Data** or **SwiftData** | SwiftData (iOS 17+) is cleaner; Core Data (iOS 15+) for wider compat |
| Persistence (cloud) | **CloudKit** | Free, built-in, uses the user's iCloud вҖ” no backend to maintain |
| Auth | **Sign in with Apple** | Required for CloudKit private database access |
| Pathfinding | **GameplayKit** (`GKGraph` + `GKGridGraph`) | Apple's A* implementation вҖ” no need to port ours |
| Build system | **XcodeGen** (`project.yml` вҶ’ `.xcodeproj`) | Project file generated by CI on macOS вҖ” editable from Windows, no merge conflicts |
| CI | **GitHub Actions** (`macos-14`) | Builds + tests on every push; TestFlight uploads on tagged releases |
| Deploy automation | **Fastlane** + `match` | Signing + upload |
| iOS floor | **iOS 17** | Enables SwiftData + `@Observable` + modern SwiftUI APIs; ~92% device coverage in 2026 |
| Bundle ID | **`com.bitedefense.game`** | Registered in Apple Developer portal |
| Orientation | **Portrait only** (Phase 1) | Matches current design |

## 4. Architecture Translation: JS вҶ’ Swift

Direct mapping of current systems to their native equivalents. This is the "Rosetta Stone" вҖ” every JS class has a Swift counterpart with the same name and responsibilities.

| Current JS | Native Swift equivalent | Notes |
|---|---|---|

| `EventBus.js` | `EventBus.swift` using Combine's `PassthroughSubject` | Typed events via enum `GameEvent` вҖ” safer than string-keyed |
| `GameState.js` | `GameState.swift` as `@Observable` class | SwiftUI observes for HUD updates |
| `Grid.js` | `IsoTileMap.swift` + `GKGridGraph` | 30Г—30 grid, occupancy tracked alongside pathing graph |
| `Camera.js` | `SKCameraNode` | SpriteKit native; gesture handlers update its position/scale |
| `IsoMath.js` | `IsoMath.swift` | Pure math; trivial port |
| `Renderer.js` | `GameScene.swift` + child SKNodes | SpriteKit handles drawing; we add/remove nodes |
| `TileRenderer.js` | `IsoTileMap.swift` | Pre-rendered tile texture cache |
| `BuildingRenderer.js` | `BuildingSprites.swift` | Use `SKShapeNode`/`SKSpriteNode` with procedural paths; same art style |

## 6. Rendering Approach

**Match the current hand-drawn look.** The existing procedural Canvas art (dogs, cats, buildings drawn with shapes) is the identity of the game вҖ” we don't swap to stock sprites.

Path forward:
1. **Port the procedural drawing code** from `BuildingRenderer.js` into `BuildingSprites.swift` using `SKShapeNode` with `CGPath` (direct translation of Canvas `moveTo`/`lineTo`/`arc` calls).
2. **Cache rendered textures** вҖ” each building type gets rendered once to an `SKTexture` via `SKView.texture(from:)`, then instanced. Massive perf win over re-drawing shapes every frame.
3. **Animations**: use `SKAction` sequences for state transitions (construction, upgrade flash, death). These are GPU-optimized and free vs. per-frame Canvas redraws.
4. **Effects layer**: damage numbers as short-lived `SKLabelNode` with `SKAction.move` + `.fadeOut`; hit sparks as `SKEmitterNode` particles (genuine native particles вҖ” impossible quality-wise in Canvas).
5. **Isometric projection**: store logical (col, row) on each node; position via `IsoMath.cartToIso` on spawn and on any (col, row) change. Never in `update()`.

This gets us **better than Canvas quality** while preserving the art style.

### Local: SwiftData

```swift
@Model

  @Relationship(deleteRule: .cascade) var troops: [SavedTroop]

```

### Cloud: CloudKit

- Container: `iCloud.com.bitedefense.game`
- Private database (user's personal iCloud), `Saves` record type mirroring the SwiftData schema
- Sync model: on app launch, fetch cloud snapshot; compare timestamps; take newer; continue saving both
- **No backend required.** CloudKit is a gift from Apple for this exact use case.

### Migration from browser saves
Not applicable вҖ” the App Store version is a fresh install, users can't carry over `localStorage`. We document this clearly in the App Store "What's New" and let early testers start fresh.

## 8. Mobile-Native Polish That Wasn't Possible in the Web Build

Things the native port unlocks:

- **True 120fps** on ProMotion devices (iPhone 13 Pro+, all Pro iPads)
- **Native haptics** with tuned patterns per event: light tick on tap, medium on placement, heavy thud on wave start, success pattern on victory, warning pattern on HQ damage. Uses `CoreHaptics` for custom patterns beyond the basic three impact styles.
- **Background audio-ducking aware music** via `AVAudioSession`
- **SF Symbols** for icons where appropriate (resource icons, action buttons)
- **Dynamic Type** support in modals so accessibility-size users aren't cut off
- **Live Activities** (Phase 2) вҖ” wave-in-progress shown on Lock Screen / Dynamic Island
- **Game Center** (Phase 2) вҖ” wave streak leaderboards, achievements
- **Haptic feedback synced to SKEmitterNode particle bursts** вҖ” viscerally satisfying in a way Canvas never is
- **Launch in < 1 second cold-start** vs. ~2вҖ“3s for the WebView version

## 9. Port Plan вҖ” Ordered Milestones

Each milestone is independently verifiable. The JS version stays playable in the browser at every step so you can A/B behavior.

### M1 вҖ” Xcode project skeleton вң…
- `ios/project.yml` (XcodeGen spec) вҖ” iOS 17 floor, portrait, bundle `com.bitedefense.game`
- SwiftUI `BiteDefenseApp` + `ContentView` hosting a `SpriteView`
- Blank `GameScene` with grass-green background and tappable "Hello Bite Defense" label
- `.github/workflows/ios-build.yml` runs `xcodegen generate` + `xcodebuild` on every push to `ios/**`, uploads a Simulator screenshot as an artifact (so we can verify visually from Windows)
- `BiteDefenseTests/` with one smoke test confirming Constants compile

**Done =** GHA build is green, screenshot artifact shows the grass scene with the label.

### M2 вҖ” Isometric tile map + camera вң…
- `IsoMath.swift` вҖ” flat top-down grid math (32px squares, matching the JS reference; "iso" naming kept for plan parity)
- `EventBus.swift` вҖ” Combine `PassthroughSubject<GameEvent>` with typed cases
- `IsoTileMap.swift` вҖ” 30Г—30 grass tiles using 4 cached `SKTexture`s (one per grass shade), seeded per-tile color match
- `InputHandler.swift` вҖ” UIKit pan/pinch/tap on the `SKView`, drives `SKCameraNode`, emits `tileTapped` + `cameraMoved`
- `GameScene` hosts camera + tile map + a debug HUD label (zoom / cam position / last tapped tile)
- Unit tests for `IsoMath` round-trip and tile picking

**Done =** Simulator screenshot shows the green 30Г—30 grid; you can pan, pinch-zoom, and the debug label updates with the tapped tile's (col, row).

### M3 вҖ” Static building art вң…
- `BuildingConfig.swift` вҖ” `BuildingType` enum + per-type `BuildingDef` (footprint, palette, emoji, max level). M4 will add cost/build-time/HP arrays.
- `BuildingSprites.swift` вҖ” bakes shadow + body + border + top-highlight band into a single cached `SKTexture` per type
- `Building.swift` (`SKNode` subclass) вҖ” body sprite + emoji label + level badge; positioned via `IsoMath.cartToWorld`
- `GameScene.addDemoBuildings(in:)` hardcodes all 7 building types so the screenshot artifact verifies parity
- Tests cover footprint values + every type having a definition

**Done =** Simulator screenshot shows HQ, Training Camp, Fort, Water Well, Milk Farm, Archer Tower, and a 3-block Wall row in the center of the grid.

### M4 вҖ” Building system + placement flow вң…
- `Grid.swift` вҖ” 30Г—30 occupancy table (separate from visual `IsoTileMap`)
- `GameState.swift` вҖ” `@Observable`: water/milk/dogCoins, `[BuildingModel]`, HQ-derived storage caps
- `GameCoordinator.swift` вҖ” top-level orchestrator with `placement` + `selectedBuildingId` UI state, drives everything from tile taps
- `BuildingSystem.swift` вҖ” `canPlace` / `place` / `move` / `remove` / `upgrade`; uniqueness + level-gating + half-refund-on-delete; coin vs. flex-resource upgrades
- Extended `BuildingConfig` with full `costs[]` / `buildTime[]` / `upgradeCoinCost[]` arrays per type
- `EventBus` extended with `buildingPlaced/Moved/Removed/Upgraded` cases; `GameScene` listens and syncs `Building` SKNodes via a `[id: Building]` map
- Placement preview: green/red SKShapeNode covering the candidate footprint; updates each frame
- SwiftUI panels: `HUDView` (top), `StorePanel` (bottom strip), `PlacementConfirmTray` (Pay Water / Pay Milk / Cancel), `BuildingInfoPanel` (Move / Upgrade / Delete)
- Tap an existing building вҶ’ info panel; tap Move вҶ’ re-enters placement mode with `movingId`; confirm relocates without re-paying

- `TrainingSystem`: queue ordering, Fort capacity math

- Smoke test: launch вҶ’ place HQ-adjacent water well вҶ’ verify HUD increments
- Smoke test: queue a soldier вҶ’ verify appears in Fort count
- Smoke test: start a wave with 0 troops вҶ’ no-troops card shown

## 12. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Scope creep вҖ” 6 weeks turns into 12 | High | Strict milestone-by-milestone delivery; each one is shippable |
| Native art doesn't match the hand-drawn web feel | Medium | Port procedural drawing code verbatim; same CGPath ops as Canvas `Path2D` |
| Pathfinding perf regression switching to GKGridGraph | Low | Benchmarked on full 30Г—30 with 50 entities; Apple's impl is optimized C |
| SpriteKit deprecation rumors | Low | Still actively maintained; used by Apple's own sample games; Metal-backed |
| CloudKit sync edge cases (multi-device conflicts) | Medium | Timestamp-based last-write-wins is fine for single-player; rare conflicts |
| Fastlane match first-setup friction | High | Budget half a day; run once manually before wiring to CI |
| iOS 16 floor loses some users | Low | Covers 97%+ in 2026; drop iOS 15 reduces SwiftUI workarounds substantially |
| App Review rejection for unclear gameplay | Low | Submit thorough review notes + a 30s gameplay video |
| Privacy manifest requirements changing | Medium | Declare zero tracking; only required-reason APIs (UserDefaults, FileTimestamp) вҖ” standard for a game |

## 13. Parallel Track: Keep the Web Version

The web version (`src/`, `index.html`, `style.css`) stays playable throughout the port. Reasons:
- **Reference implementation** for balance tuning and UX decisions вҖ” if the native version "feels different" we can A/B immediately
- **Prototype environment** вҖ” new features get built in JS first, validated, then ported
- **Demo/marketing** вҖ” shareable URL for people without iPhones
- **Contingency** вҖ” if the native port stalls, we haven't lost the game

After App Store launch, the web version continues as the dev sandbox for upcoming features.
