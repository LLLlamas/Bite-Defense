# Bite Defense — Native iOS Port Plan (Swift + SpriteKit)

**Goal:** ship Bite Defense as a fully native iOS app built in Swift with SpriteKit for rendering and game logic. Target TestFlight first, then public App Store release. The existing JS/Canvas version becomes a reference implementation — design, balance, and feel stay, but all code is rewritten natively.

**Why native over Capacitor:** tighter 60/120fps rendering, real Metal acceleration, native haptics/animations with zero bridge latency, smaller binary, better battery life, easier App Store approval ("real" apps aren't scrutinized for minimal functionality), and a foundation for future features like Game Center, iCloud sync, and iPad-optimized layouts.

**Apple Developer Program:** ✅ enrolled.

**Reference implementation:** the current JS codebase in `src/` stays in-repo as the source of truth for game design, balance numbers, and UX decisions. No functional change to it during the port.

## 1. Technology Stack

| Layer | Choice | Why |
|---|---|---|
| Language | **Swift 5.9+** | Apple's first-class language; great tooling |
| Game engine | **SpriteKit** | Apple-native 2D engine, perfect for tile-based isometric games, built-in physics and actions, zero dependencies |
| UI chrome (HUD, panels, modals) | **SwiftUI** | Modern declarative UI, composes cleanly with SpriteKit via `SpriteView` |
| Persistence (local) | **Core Data** or **SwiftData** | SwiftData (iOS 17+) is cleaner; Core Data (iOS 15+) for wider compat |
| Persistence (cloud) | **CloudKit** | Free, built-in, uses the user's iCloud — no backend to maintain |
| Auth | **Sign in with Apple** | Required for CloudKit private database access |
| Pathfinding | **GameplayKit** (`GKGraph` + `GKGridGraph`) | Apple's A* implementation — no need to port ours |
| Build system | **XcodeGen** (`project.yml` → `.xcodeproj`) | Project file generated by CI on macOS — editable from Windows, no merge conflicts |
| CI | **GitHub Actions** (`macos-14`) | Builds + tests on every push; TestFlight uploads on tagged releases |
| Deploy automation | **Fastlane** + `match` | Signing + upload |
| iOS floor | **iOS 17** | Enables SwiftData + `@Observable` + modern SwiftUI APIs; ~92% device coverage in 2026 |
| Bundle ID | **`com.bitedefense.game`** | Registered in Apple Developer portal |
| Orientation | **Portrait only** (Phase 1) | Matches current design |

## 4. Architecture Translation: JS → Swift

Direct mapping of current systems to their native equivalents. This is the "Rosetta Stone" — every JS class has a Swift counterpart with the same name and responsibilities.

| Current JS | Native Swift equivalent | Notes |
|---|---|---|
| `GameLoop.js` | `GameScene.update(_:)` (SpriteKit's built-in loop) | SpriteKit provides the loop; we don't own it |
| `EventBus.js` | `EventBus.swift` using Combine's `PassthroughSubject` | Typed events via enum `GameEvent` — safer than string-keyed |
| `GameState.js` | `GameState.swift` as `@Observable` class | SwiftUI observes for HUD updates |
| `Grid.js` | `IsoTileMap.swift` + `GKGridGraph` | 30×30 grid, occupancy tracked alongside pathing graph |
| `Camera.js` | `SKCameraNode` | SpriteKit native; gesture handlers update its position/scale |
| `IsoMath.js` | `IsoMath.swift` | Pure math; trivial port |
| `Renderer.js` | `GameScene.swift` + child SKNodes | SpriteKit handles drawing; we add/remove nodes |
| `TileRenderer.js` | `IsoTileMap.swift` | Pre-rendered tile texture cache |
| `BuildingRenderer.js` | `BuildingSprites.swift` | Use `SKShapeNode`/`SKSpriteNode` with procedural paths; same art style |
| `InputHandler.js` | `InputHandler.swift` | `UITapGestureRecognizer`, `UIPinchGestureRecognizer`, `UIPanGestureRecognizer` on the `SKView` |
| `BuildingSystem.js` | `BuildingSystem.swift` | Same API: `placeBuilding`, `moveBuilding`, `canPlace` |
| `ResourceSystem.js` | `ResourceSystem.swift` | Timer-driven generation; publishes via Combine |
| `TrainingSystem.js` | `TrainingSystem.swift` | Queue-based; SwiftUI binds to queue for progress bars |
| `PathfindingSystem.js` | `PathfindingSystem.swift` using `GKGridGraph` | Apple's A* — delete ours |
| `WaveSystem.js` | `WaveSystem.swift` | Phase machine: BUILDING → PRE_BATTLE → BATTLE → complete/fail |
| `CombatSystem.js` | `CombatSystem.swift` | Target selection, damage ticks, death handling |
| `TroopPlacementSystem.js` | `TroopPlacementSystem.swift` | Manual positioning during PRE_BATTLE |
| `UIManager.js` | Fragmented into SwiftUI views (see UI/ folder) | God-object goes away; views bind directly to `@Observable` state |
| `FloatingResources.js` | `FloatingResource.swift` SwiftUI view + `.transition` | Matches existing fly-to-HUD animation using SwiftUI's matched geometry effect |
| `BuildingConfig.js` / `TroopConfig.js` | `BuildingConfig.swift` / `TroopConfig.swift` | Static enums + structs; compile-time safety |
| `Constants.js` | `Constants.swift` | |
| `localStorage` save | `SaveManager.swift` → SwiftData + CloudKit | Migration path from zero (fresh native install) |

SwiftUI wraps SpriteKit via `SpriteView(scene: gameScene)`. Events flow: SpriteKit emits via EventBus → SwiftUI views update via `@Observable` bindings. User input on HUD/panels goes directly to `GameCoordinator`; taps on the battlefield go through `GameScene.touchesEnded`.

## 6. Rendering Approach

**Match the current hand-drawn look.** The existing procedural Canvas art (dogs, cats, buildings drawn with shapes) is the identity of the game — we don't swap to stock sprites.

Path forward:
1. **Port the procedural drawing code** from `BuildingRenderer.js` into `BuildingSprites.swift` using `SKShapeNode` with `CGPath` (direct translation of Canvas `moveTo`/`lineTo`/`arc` calls).
2. **Cache rendered textures** — each building type gets rendered once to an `SKTexture` via `SKView.texture(from:)`, then instanced. Massive perf win over re-drawing shapes every frame.
3. **Animations**: use `SKAction` sequences for state transitions (construction, upgrade flash, death). These are GPU-optimized and free vs. per-frame Canvas redraws.
4. **Effects layer**: damage numbers as short-lived `SKLabelNode` with `SKAction.move` + `.fadeOut`; hit sparks as `SKEmitterNode` particles (genuine native particles — impossible quality-wise in Canvas).
5. **Isometric projection**: store logical (col, row) on each node; position via `IsoMath.cartToIso` on spawn and on any (col, row) change. Never in `update()`.

This gets us **better than Canvas quality** while preserving the art style.

## 8. Mobile-Native Polish That Wasn't Possible in the Web Build

Things the native port unlocks:

- **True 120fps** on ProMotion devices (iPhone 13 Pro+, all Pro iPads)
- **Native haptics** with tuned patterns per event: light tick on tap, medium on placement, heavy thud on wave start, success pattern on victory, warning pattern on HQ damage. Uses `CoreHaptics` for custom patterns beyond the basic three impact styles.
- **Background audio-ducking aware music** via `AVAudioSession`
- **SF Symbols** for icons where appropriate (resource icons, action buttons)
- **Dynamic Type** support in modals so accessibility-size users aren't cut off
- **Live Activities** (Phase 2) — wave-in-progress shown on Lock Screen / Dynamic Island
- **Game Center** (Phase 2) — wave streak leaderboards, achievements
- **Haptic feedback synced to SKEmitterNode particle bursts** — viscerally satisfying in a way Canvas never is
- **Launch in < 1 second cold-start** vs. ~2–3s for the WebView version

## 10. Testing Strategy

### Unit tests (XCTest)
- All systems get unit-tested independently
- `BuildingSystem`: placement validation, movement, deletion-refund math
- `ResourceSystem`: generation rates, cap enforcement
- `TrainingSystem`: queue ordering, Fort capacity math
- `CombatSystem`: target selection, damage calc
- `PathfindingSystem`: known-map regression tests
- `WaveSystem`: phase transitions

### Snapshot tests
- `BuildingSprites` each type rendered to texture → compared against committed reference PNG
- Catches accidental visual regressions during art iteration

### UI tests (XCUITest)
- Smoke test: launch → place HQ-adjacent water well → verify HUD increments
- Smoke test: queue a soldier → verify appears in Fort count
- Smoke test: start a wave with 0 troops → no-troops card shown

### Manual device matrix
- iPhone 12 mini (smallest recent)
- iPhone 15 Pro Max (largest + ProMotion)
- iPhone SE 3rd gen (oldest target, smallest screen)

### Perf budget
- Stable 60fps during Nightmare difficulty with 50+ entities on screen
- Cold launch < 1.2s on iPhone 11
- Memory < 150MB resident during battle
- Battery drain < 8%/hour during active play

## 12. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Scope creep — 6 weeks turns into 12 | High | Strict milestone-by-milestone delivery; each one is shippable |
| Native art doesn't match the hand-drawn web feel | Medium | Port procedural drawing code verbatim; same CGPath ops as Canvas `Path2D` |
| Pathfinding perf regression switching to GKGridGraph | Low | Benchmarked on full 30×30 with 50 entities; Apple's impl is optimized C |
| SpriteKit deprecation rumors | Low | Still actively maintained; used by Apple's own sample games; Metal-backed |
| CloudKit sync edge cases (multi-device conflicts) | Medium | Timestamp-based last-write-wins is fine for single-player; rare conflicts |
| Fastlane match first-setup friction | High | Budget half a day; run once manually before wiring to CI |
| iOS 16 floor loses some users | Low | Covers 97%+ in 2026; drop iOS 15 reduces SwiftUI workarounds substantially |
| App Review rejection for unclear gameplay | Low | Submit thorough review notes + a 30s gameplay video |
| Privacy manifest requirements changing | Medium | Declare zero tracking; only required-reason APIs (UserDefaults, FileTimestamp) — standard for a game |

## 13. Parallel Track: Keep the Web Version

The web version (`src/`, `index.html`, `style.css`) stays playable throughout the port. Reasons:
- **Reference implementation** for balance tuning and UX decisions — if the native version "feels different" we can A/B immediately
- **Prototype environment** — new features get built in JS first, validated, then ported
- **Demo/marketing** — shareable URL for people without iPhones
- **Contingency** — if the native port stalls, we haven't lost the game

After App Store launch, the web version continues as the dev sandbox for upcoming features.
